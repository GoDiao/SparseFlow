# Handoff

## Current State

SparseFlow is an open-source research project for tiered-memory inference on
sparse expert models (MoE). The project is inspired by
`JustVugg/colibri`, but its core should remain model-independent.

The current experiment target is `Qwen/Qwen3.6-35B-A3B`. The project is not
intended to become a Qwen-only wrapper.

The main hypothesis is:

```text
dense/router/shared weights resident in RAM
        +
routed experts streamed from NVMe/SSD on demand
        +
hot expert cache, heat policy, and prefetch
```

The repository is public at <https://github.com/GoDiao/SparseFlow>.
The default branch is `main`, and the local branch tracks `origin/main`.

This is a good fit for testing expert streaming and caching. It does not yet
prove that a complete Qwen3.6 generation runtime will be fast or comfortable.

## Key Design Decisions

### Model-independent core

The core should own:

- `ShardIndex`: safetensors header indexing without loading payloads.
- `ModelFootprintAnalyzer`: category sizes, per-layer sizes, and read estimates.
- `TierPlanner`: placement decisions based on RAM, VRAM, disk, and context.
- `ExpertLocator`: logical expert to physical tensor slice mapping.
- `ExpertCache`: CPU/GPU hot tier and LRU behavior.
- `HotTierPolicy`: usage count, heat decay, hysteresis, and promotion/eviction.
- Prefetch and route-trace simulation.

### Qwen3.6 adapter

Qwen3.6 routed experts are stored in fused tensors, principally:

```text
model.language_model.layers.<layer>.mlp.experts.gate_up_proj
model.language_model.layers.<layer>.mlp.experts.down_proj
```

An expert is therefore a byte range inside a tensor, not necessarily one
standalone file. `ExpertLocator` must derive the slice from shape, dtype,
expert count, and the actual tensor layout. This is the first important
correctness boundary before implementing a loader or cache.

### Current non-goals

- Do not implement the complete Qwen3.6 forward pass first.
- Do not promise a production chat server or high-throughput serving.
- Do not make vision, MTP, or every Gated DeltaNet detail a prerequisite for
  the first storage/cache experiments.
- Do not couple the core API to Qwen3.6 naming rules.

## Current Objective

Use the remote big-memory CPU machine to validate the storage-side hypothesis:

1. Locate one Qwen3.6 expert inside a fused safetensors tensor.
2. Read only that expert's byte ranges and verify the byte count.
3. Measure cold-read and warm-read latency from the remote disk.
4. Implement a CPU LRU cache keyed by `(layer, expert_id)`.
5. Compare cache capacity, hit rate, memory use, and effective read volume.

The planned experiment machine has about 128GB RAM and a sufficiently large
NVMe/SSD volume. GPU is optional for this phase; it becomes useful later for
testing a GPU hot tier.

### 3. Implement `ExpertLocator`

The first implementation should:

- accept `(layer, expert_id)`;
- find both `gate_up_proj` and `down_proj` fused tensors;
- calculate byte offsets from tensor metadata;
- return shard, offset, length, dtype, and logical shape;
- validate that the calculated range stays within the tensor payload.

Before reading a large range, verify the result with small reads and exact
shape/byte-size assertions. Do not assume the expert dimension order without
checking the actual Qwen3.6 metadata.

### 4. Add `expert-stat` and `expert-load`

Planned commands:

```bash
python -m sparseflow expert-stat <model> --layer 0 --expert 17
python -m sparseflow expert-load <model> --layer 0 --expert 17
```

The first version only needs to report and verify slices. It does not need to
perform a model forward pass.

### 5. Add cache and route simulation

- Implement a per-layer CPU LRU cache keyed by `(layer, expert_id)`.
- Add usage counts, heat decay, and hysteresis after basic LRU works.
- Simulate batch-level unique expert union for different batch sizes.
- Record hit rate, cold bytes per token, and the estimated disk bandwidth
  requirement.
- Add async prefetch only after measurements show it is useful.

## Important Risks

- Qwen3.6 combines Gated DeltaNet, Gated Attention, MoE, Vision, and MTP;
  complete runtime implementation is substantially harder than a plain
  Transformer MoE.
- The `6.00 MiB` expert estimate does not capture every layout, alignment,
  quantization, or kernel cost.
- The theoretical `1.88 GiB` cold read per token makes disk bandwidth and
  cache hit rate central performance variables.
- 128GB RAM is suitable for analysis and streaming prototypes, but does not
  imply comfortable generation speed.
- Verify model and dependency licenses before distributing weights, converted
  artifacts, or model-specific adapters.

## Operational Constraints

- Do not commit model weights, downloaded caches, virtual environments, or
  remote credentials.
- Keep local caches and temporary files on `E:`; keep remote caches on the
  large data disk.
- Prefer `uv` for Python environments and dependency installation.
- Add a small reproducible test for each new CLI or core algorithm.
- Record real hardware, filesystem, latency, cache-hit, and read-volume data.
- Update this file after each meaningful experiment or design decision.
