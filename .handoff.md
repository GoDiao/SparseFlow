# SparseFlow 项目交接记录

更新时间：2026-07-13

## 项目目标

SparseFlow 是一个面向稀疏专家模型（MoE）的分层内存推理实验项目，借鉴 `JustVugg/colibri` 的思路：

- 将每次都要访问的 dense、router、shared expert 等权重放在 RAM 或更快的存储层。
- 将 routed experts 保存在本地 NVMe/SSD，按路由结果按需读取。
- 使用 RAM/VRAM 热缓存、LRU、热度统计和预取降低磁盘读取开销。
- 保持模型无关的核心模块，并通过 adapter 适配不同的 MoE 模型。

当前首个实验对象是 `Qwen/Qwen3.6-35B-A3B`，但项目不会绑定 Qwen。

### 本地模型体检结果

本地模型约 `66.97 GiB`，共有 26 个 safetensors 文件。初步统计：

- routed experts：约 `60.00 GiB`
- dense resident estimate：约 `6.97 GiB`
- 典型 routed expert：约 `6.00 MiB`
- 每层 routed experts：约 `1.50 GiB`
- 40 层每层一个 cache slot：约 `240 MiB`
- 每 token 理论冷读量：约 `1.88 GiB`

这些数字说明它适合验证“dense 常驻 + expert 按需读取 + 热 expert 缓存”，但完整推理仍需要进一步实现和实测。

## 当前关键设计

### 模型无关核心

- `ShardIndex`：读取 safetensors header，建立 tensor 名称到 shard、offset、size、dtype、shape 的索引。
- `ModelFootprintAnalyzer`：统计模型各类别权重占用和读取量。
- `TierPlanner`：根据 RAM、VRAM、磁盘和上下文预算生成分层方案。
- 后续核心：`ExpertLocator`、`ExpertCache`、`HotTierPolicy`、prefetch。

### Qwen3.6 adapter

Qwen3.6 的 routed experts 采用 fused tensor，主要形式是：

- `model.language_model.layers.<layer>.mlp.experts.gate_up_proj`
- `model.language_model.layers.<layer>.mlp.experts.down_proj`

因此 expert 不是简单的“一个 expert 对应一个文件”，而是需要在 fused tensor 内根据 expert 维度计算切片。这是后续实现最重要的模型适配点之一。

### 当前非目标

- 暂不从零实现完整 Qwen3.6 forward。
- 暂不承诺完整聊天体验或高吞吐服务。
- 暂不优先处理 vision、MTP 和所有复杂的 Gated DeltaNet 细节。
- 暂不把项目设计成只支持 Qwen3.6 的 wrapper。

## 当前工作目标

在远程大内存 CPU 试验机上，验证 SparseFlow 的核心假设：

1. 能否从 fused safetensors tensor 中精确定位并读取单个 expert。
2. expert cache 在不同容量下的命中率和内存占用。
3. 冷 expert 读取、热 expert 命中和 page cache 对性能的影响。
4. 128GB RAM + NVMe 是否足以支撑 dense 常驻和 expert streaming 实验。

远程机器此前计划采用约 128GB RAM、较大 SSD；GPU 不是第一阶段的必要条件。

## 下一步执行顺序

### 1. 远程环境确认

在试验机执行并记录：

```bash
df -h
free -h
lscpu
lsblk
nvidia-smi || true
```

模型、ModelScope/Hugging Face 缓存、Python/uv 环境和 npm 缓存应放在大容量数据盘，不要挤占系统盘。

### 2. 拉取并运行当前版本

```bash
git clone git@github.com:GoDiao/SparseFlow.git
cd SparseFlow
export PYTHONPATH="$PWD/src"
python -m unittest discover -s tests -p 'test_*.py'
python -m sparseflow inspect /data/model/Qwen3.6-35B-A3B
python -m sparseflow plan /data/model/Qwen3.6-35B-A3B --ram 96 --ctx 4096
```

如需安装开发环境，优先使用 `uv`，并将缓存指定到数据盘。

### 3. 实现 ExpertLocator

新增模块负责：

- 根据 `(layer, expert_id)` 查找 Qwen3.6 的 fused expert tensor。
- 根据 shape、dtype 和 expert 维度计算 byte range。
- 同时定位 `gate_up_proj` 和 `down_proj` 两组权重。
- 返回可验证的 slice 元数据，不立即加载完整 tensor。

必须先用小范围读取和 shape/byte-size 校验验证 offset 计算，避免把 fused tensor 的维度顺序假设错。

### 4. 实现 expert-stat / expert-load

计划增加 CLI：

```bash
python -m sparseflow expert-stat <model> --layer 0 --expert 17
python -m sparseflow expert-load <model> --layer 0 --expert 17
```

第一版只需要输出 slice 的 shard、offset、长度、dtype、shape，并验证读取字节数；不要求接入完整推理。

### 5. 实现 cache 和路由模拟

- 先实现按 `(layer, expert_id)` 组织的 CPU LRU cache。
- 再加入 usage count、heat decay、hysteresis 等热度策略。
- 增加 batch 内 unique expert union 的模拟工具。
- 记录 cache 命中率、每 token 冷读量、理论带宽瓶颈。

## 重要风险

- Qwen3.6 是混合架构，包含 Gated DeltaNet、Gated Attention、Vision 和 MTP；完整 runtime 的难度明显高于普通 Transformer MoE。
- `6.00 MiB` 是典型 expert 权重大小，实际读取还会受到 tensor 布局、量化格式、对齐和计算 kernel 影响。
- 每 token 理论冷读量约 `1.88 GiB`，若 cache 命中率低，SSD 带宽会成为主要瓶颈。
- 128GB RAM 能支撑分析和 streaming 原型，但不代表可以获得舒适的完整生成速度。
- 需要确认模型权重许可证、ModelScope 下载内容和后续发布的代码/转换产物符合各自许可要求。
